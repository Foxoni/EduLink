## Project Overview

EduLink is a Flask-based school management web application with role-based access control (RBAC) for three user types: Admin, Professor, and Student.

## Running the Application

**With Docker (recommended):**
```bash
docker-compose up -d
# App: http://localhost:5001 | PHPMyAdmin: http://localhost:8080
docker exec gcs_flask_app python seed.py  # seed test data
```

**Locally:**
```bash
pip install -r requirements.txt
cp .env.example .env  # then fill in MySQL credentials
mysql -u dev_user -p EduLink < init.sql
python seed.py
python app.py  # runs on http://localhost:5000
```

## Linting & CI checks

```bash
flake8 .         # linting (E9, F63, F7, F82 are hard errors)
pip-audit        # dependency security audit
safety check     # additional dependency CVE scan
```

The GitHub Actions pipeline (`.github/workflows/devsecops.yml`) runs linting + security scans on every push/PR to `main`, then builds the Docker image.

## Architecture

The app uses **Flask blueprints** organized by role:

- `blueprints/auth/` — login/logout
- `blueprints/admin/` — user, class, and schedule management (role_id=1)
- `blueprints/prof/` — teacher dashboard (role_id=2)
- `blueprints/eleve/` — student dashboard (role_id=3)

**Authentication & Authorization:**
- Session-based auth with 1-hour TTL
- `decorators.py` provides `@login_required` and `@role_required(role_id)` — apply these to all protected routes
- Session is regenerated on login to prevent session fixation

**Database (`db.py`):**
- Raw MySQL via `mysql-connector-python` — no ORM
- Always use parameterized queries (`%s` placeholders) — never string-interpolate user input into SQL
- Schema defined in `init.sql`; test data seeded by `seed.py`

**Key tables:** `Utilisateurs` (unified user table with role_id), `Classes`, `Emploi_du_temps` (schedule), `Evaluation`, `Notes`, `Prof_Classe` (many-to-many prof↔class)

**Security headers** are set globally in `app.py` via `@app.after_request`: CSP, X-Frame-Options, X-Content-Type-Options, Referrer-Policy. CSRF protection is via Flask-WTF.

## Important: Two Entry Points

- `app.py` — current architecture with blueprints (use this for development)
- `admin.py` — legacy standalone routes (currently referenced in Dockerfile; still partially active)

When adding admin features, check both files to avoid duplication.
